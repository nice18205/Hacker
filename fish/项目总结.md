# 项目总结

## 1. 项目定位

这是一个基于浏览器运行的 3D 交互式钓鱼小游戏，核心卖点不是传统键鼠控制，而是“手机陀螺仪甩杆 + 触屏替代操作 + 鱼获处理”的完整体验闭环。

项目目标可以概括为三点：

- 用 `Three.js` 在单页面中搭建可实时渲染的 3D 钓鱼场景。
- 在移动端利用 `DeviceOrientationEvent` 和 `DeviceMotionEvent` 实现后仰蓄力、前甩抛竿、上扬收杆、挥切处理鱼获等动作映射。
- 通过一个极简 Python 启动脚本提供本地 `HTTP/HTTPS` 服务，解决手机访问和传感器权限问题。

从产品形态上看，这不是“工程化前端项目”，而是一个偏原型/演示性质的独立 Demo，强调可直接打开、快速试玩和跨设备体验。

## 2. 项目结构

项目结构非常精简，只有 4 个关键组成部分：

- `demo.html`
  - 核心前端文件，包含页面结构、样式、Three.js 场景构建、游戏状态机、输入处理、渲染循环。
- `demo.py`
  - 本地服务启动器，支持自动挑选端口、输出局域网地址、可选 HTTPS、自签名证书生成。
- `vendor/three.module.js`
  - 本地 vendored 的 Three.js 模块文件，避免依赖 npm 或 CDN。
- `.gitignore`
  - 忽略证书目录、缓存和临时日志文件。

这意味着项目没有：

- 构建工具
- 包管理配置
- 模块拆分
- 自动化测试
- 独立资源目录

优点是上手门槛低；缺点是维护成本会随着功能增长迅速上升。

## 3. 技术栈与依赖

### 前端

- 原生 `HTML + CSS + JavaScript ES Module`
- `Three.js` 本地模块引入
- 浏览器原生传感器 API
  - `DeviceOrientationEvent`
  - `DeviceMotionEvent`
- 浏览器原生震动 API
  - `navigator.vibrate`

### 后端/本地服务

- `Python 3`
- 标准库：
  - `http.server`
  - `ssl`
  - `socket`
  - `subprocess`
  - `webbrowser`

### 外部环境依赖

- 若启用 HTTPS，需要本机安装 `openssl`
- 若在手机端使用陀螺仪，通常必须通过 `HTTPS` 安全上下文访问页面

## 4. 运行方式

### 桌面调试

可直接使用 HTTP 模式启动：

```bash
python demo.py --http
```

适用于：

- 桌面浏览器查看画面
- 鼠标/触屏拖拽调试
- 不依赖陀螺仪时的基础玩法验证

### 手机陀螺仪体验

推荐直接启动 HTTPS：

```bash
python demo.py
```

运行后脚本会：

- 自动选择端口
- 自动输出本机局域网访问地址
- 自动尝试打开本地浏览器
- 在首次需要时生成自签名证书

手机端访问时需要注意：

- 先接受本地证书告警
- 再点击页面里的“陀螺仪”
- 某些浏览器或系统仍可能拦截传感器权限

## 5. 整体架构

虽然代码都写在一个文件里，但逻辑上可以清晰划分成 5 层。

### 5.1 视图层

由 HTML 和 CSS 组成，主要包括：

- 全屏 `canvas` 作为 Three.js 渲染目标
- 覆盖式 HUD 面板
- 全屏触摸层 `touch-zone`
- 视觉反馈元素
  - 咬钩闪光
  - 抛竿提示
  - 操作按钮组

这一层负责信息展示和基础交互承接，不承担复杂业务判断。

### 5.2 渲染层

Three.js 负责：

- 渲染器初始化
- 场景、相机、灯光、雾效
- 水面、岸边、木板、山体
- 鱼竿、浮漂、鱼群、处理台
- 水波和水滴等即时特效

场景不是资源驱动，而是全部由程序动态创建的“参数化几何体场景”。

### 5.3 领域状态层

项目把几乎所有游戏运行态收敛在 `game` 对象中，包括：

- 主状态
- 分数
- 蓄力
- 瞄准
- 陀螺仪校准数据
- 鱼获列表
- 处理模式
- 粒子与水波特效集合

这是整个项目的“单一状态源”。

### 5.4 输入适配层

输入来源分为两类：

- 指针输入
  - 按下、拖动、抬起、点击
- 传感器输入
  - 方向角
  - 加速度
  - 旋转速度

代码把两类输入都映射到同一套游戏行为上，例如：

- 蓄力
- 抛竿
- 收杆
- 处理鱼获

因此桌面和手机共用同一套核心玩法逻辑。

### 5.5 驱动循环层

`animate()` 中持续执行：

- 更新鱼竿
- 更新浮漂
- 更新鱼群
- 更新水面
- 更新粒子特效
- 更新相机
- 更新 HUD
- 调用 Three.js 渲染

这是一个典型的帧循环驱动型前端小游戏结构。

## 6. 关键玩法闭环

项目的核心玩法不是单步钓鱼，而是由两段循环组成。

### 6.1 钓鱼循环

完整链路如下：

1. 玩家蓄力
2. 玩家抛竿
3. 浮漂飞行并落水
4. 进入等待咬钩阶段
5. 鱼接近浮漂时会加速触发咬钩
6. 玩家在窗口期内收杆
7. 成功后生成鱼获、增加分数、把鱼放到木台上

### 6.2 处理循环

玩家钓到鱼后，可以进入“处理模式”：

1. 选择一条未处理鱼获
2. 通过快速下划或挥动设备执行切鱼动作
3. 累计切刀次数
4. 达到该鱼要求的刀数后转化为料理份数
5. 自动切换到下一条鱼，直到全部处理完成

这让项目从“钓鱼演示”升级为“钓鱼 + 加工”的双阶段小游戏。

## 7. 状态机理解

项目的主状态围绕 `game.state` 展开，主要状态包括：

- `idle`
  - 待机状态，可开始蓄力或进入处理模式
- `charge`
  - 蓄力状态，可继续拉升力度并调整瞄准方向
- `cast`
  - 抛竿飞行状态，浮漂做抛物线运动
- `wait`
  - 浮漂落水后的等待状态
- `bite`
  - 咬钩窗口期，等待玩家收杆
- `caught`
  - 已钓中状态，短暂停留后回到待机

此外还有一个与主状态并行的子模式：

- `processMode`
  - 为 `true` 时，表示当前进入鱼获处理流程

这种设计的特点是：

- 主状态机负责“钓鱼”
- 模式开关负责“处理”
- 两条逻辑共享同一套输入渠道，但在事件入口做分流

这是一个简单但有效的状态组织方式。

## 8. 3D 场景设计

场景构建采用纯代码生成，没有贴图资源依赖，主要特点如下：

- 天空使用自定义 Shader 做上下渐变
- 水面使用高分段平面加顶点位移模拟波浪
- 河岸、沙地、木板和远景山体都由基础几何体组合而成
- 鱼竿、浮漂和鱼使用组合几何体拼装
- 水花、水滴、水波通过短生命周期对象实现

这种实现方式的优点：

- 资源轻
- 易部署
- 无需美术资源管线

不足：

- 可视表现上限有限
- 代码体积会随模型复杂度线性增长

## 9. 输入与交互映射

### 9.1 触屏/鼠标操作

桌面和无传感器设备主要通过指针事件游玩：

- `pointerdown`
  - 开始蓄力，或在处理模式下记录切鱼起点
- `pointermove`
  - 调整蓄力强度和瞄准方向
- `pointerup`
  - 结束蓄力并执行抛竿，或完成一次切鱼判断
- `click`
  - 咬钩时收杆；非咬钩时按点击位置修正瞄准

### 9.2 陀螺仪操作

手机端通过两套传感器协同：

- `deviceorientation`
  - 用于获取 `beta/gamma`
  - 映射后仰蓄力和左右瞄准
- `devicemotion`
  - 用于检测加速度和角速度变化
  - 识别抛竿、收杆和切鱼动作

陀螺仪控制并不是简单阈值判断，而是经过了几个步骤：

- 先校准基线姿态
- 再判断是否完成“抬起蓄力”
- 然后在时间窗口内识别反向甩动
- 最终把识别结果映射为抛竿或处理动作

因此这一部分是整个项目最有技术含量的区域。

## 10. 鱼与得分系统

项目内置了多种鱼类，每种鱼都包含以下属性：

- `id`
- `name`
- `color`
- `finColor`
- `rarity`
- `minScale`
- `maxScale`
- `cuts`
- `value`

这些属性分别影响：

- 出现概率
- 模型大小
- 处理所需刀数
- 捕获得分
- 处理后可产出的料理份数

这使得游戏拥有最基础的数值系统和轻量稀有度设计。

## 11. Python 启动脚本的职责

`demo.py` 的职责不是业务后端，而是“本地运行环境适配器”，主要承担：

- 提供静态文件服务
- 自动寻找可用端口
- 收集本机局域网 IP
- 在 HTTPS 模式下生成自签名证书
- 输出桌面和手机可访问地址
- 设置与传感器相关的响应头

它的价值很高，因为移动端传感器体验对运行环境有明显要求：

- 仅双击 HTML 文件打开，无法稳定获得传感器权限
- 仅 HTTP 服务通常无法在手机上启用陀螺仪
- 需要局域网访问和 HTTPS 支持时，Python 脚本能直接补足这层能力

## 12. 项目优点

这个项目的完成度高于普通 Demo，主要优点如下：

- 玩法闭环完整，不只是展示场景
- 桌面和手机双输入路径兼容
- 对传感器权限、HTTPS、安全上下文有明确处理
- 视觉反馈丰富，包含提示、震动、闪光、水花、水波、落点预览
- 通过鱼获处理机制扩展了玩法深度
- 无需工程化依赖，克隆后即可直接运行

## 13. 当前问题与风险

### 13.1 代码集中度过高

所有前端逻辑都堆在 `demo.html` 中，HTML、CSS、业务状态、Three.js 创建逻辑、输入系统和渲染循环高度耦合。

直接风险：

- 功能继续增加时会迅速失控
- 定位问题难
- 复用困难
- 测试几乎无法开展

### 13.2 缺少模块边界

当前没有明确拆分：

- 场景构建模块
- 状态管理模块
- 输入适配模块
- 鱼获/数值模块
- UI/HUD 模块

后续如果新增关卡、背包、更多料理逻辑，重构成本会很高。

### 13.3 缺少错误恢复和降级细化

虽然已经对传感器权限和 HTTPS 做了提示，但仍属于轻量级提示，缺少更细的诊断手段，例如：

- OpenSSL 未安装时的明确提示
- 自签名证书生成失败时的可恢复说明
- 不同浏览器传感器差异的兼容说明

### 13.4 性能隐患

当前项目规模不大时问题不明显，但存在潜在点：

- 水面每帧逐顶点更新
- 粒子与水波对象频繁创建/销毁
- 已钓鱼获长期驻留场景
- 主循环中逻辑集中且无分层调度

在移动设备上扩大内容量后可能出现掉帧。

### 13.5 可维护性细节

存在个别“原型期痕迹”，例如：

- 常量 `PROCESS_STROKE_DISTANCE` 已定义但未实际使用
- 一些处理流程字段已存在但当前逻辑使用较少
- 说明文案和阈值高度写死在源码中

这类问题不影响当前可玩性，但会增加后续迭代摩擦。

## 14. 建议的重构方向

如果要把这个 Demo 升级为可持续维护的小项目，建议按以下优先级推进。

### 第一阶段：做结构拆分

建议把前端拆成至少 6 个模块：

- `main.js`
- `scene.js`
- `state.js`
- `input.js`
- `gameplay.js`
- `ui.js`

目标不是引入复杂框架，而是先把职责切开。

### 第二阶段：提取配置

把以下内容集中到配置对象：

- 鱼种定义
- 传感器阈值
- 抛竿参数
- 粒子参数
- UI 文案

这样更利于调参和后续玩法扩展。

### 第三阶段：补运行文档

建议新增正式 `README`，至少写清楚：

- Python 和 OpenSSL 前置条件
- HTTP 与 HTTPS 的使用差异
- 手机端推荐浏览器
- 证书告警处理方式
- 常见故障排查

### 第四阶段：增加基础测试

虽然 3D 和传感器不容易做完整自动化测试，但仍可以补：

- 纯函数/数值函数单测
- 鱼种配置合法性检查
- 状态机转换约束测试

## 15. 一句话结论

这是一个质量不错的单文件 3D 交互 Demo：前端用 Three.js 搭建了完整钓鱼与处理玩法，后端用 Python 解决了本地 HTTPS 和手机传感器访问问题。它的最大优点是“轻量但完整”，最大短板是“功能都写在一个文件里，后续演进会越来越难”。
